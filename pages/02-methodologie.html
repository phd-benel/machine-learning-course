Chapitre 2 - Méthodologie :La "Data Supply Chain" – De la Vision à l'Impact

Après avoir cadré la problématique avec la Mairie de Paris, vous devez concevoir l'architecture technique qui transformera des téraoctets de données brutes en une décision politique. Un projet de Data Science n'est pas une simple suite de scripts isolés ; c'est une chaîne de valeur logique et automatisée.

1. La Vue d'Ensemble : La Pipeline du Projet

L'ingénieur ne se contente pas d'analyser ; il construit une "usine" à données. Voici le cheminement critique :

$$\text{Problématique (Ch. 1)} \xrightarrow{\text{Phase 1}} \text{Collecte \& Ingestion} \xrightarrow{\text{Phase 2}} \text{ETL \& Enrichissement} \xrightarrow{\text{Phase 3}} \text{EDA \& Storytelling}$$

2. Phase 1 : Collecte & Ingestion – Dompter le Chaos du Data Lake

C'est ici que vous gérez la Variété et le Volume. Le but est de créer un Data Lake : un réservoir centralisé où l'on stocke les données sous leur forme native (brute).

Multimodalité et Sources :

Structurées (CSV/SQL) : Les fichiers listings.csv contenant les prix et les caractéristiques des logements.

Semi-structurées (JSON/API) : Interroger des APIs de géolocalisation pour enrichir les quartiers.

Non-structurées (Texte/Images) : Des milliers de commentaires clients et des photos de façades.

Concepts Clés :

Batch vs Temps Réel : Pour ce projet, nous travaillons en Batch (traitement par lots de fichiers historiques), mais un ingénieur doit anticiper le Temps Réel (streaming) si la Mairie voulait une alerte immédiate lors d'une nouvelle annonce suspecte.

Ingestion "Idempotente" : Votre script d'ingestion doit pouvoir être relancé dix fois sans créer de doublons. C'est la base de la fiabilité.

💡 Conseil d'expert : Ne modifiez jamais les fichiers dans le Data Lake. Gardez toujours une copie "Raw" (brute) pour pouvoir ré-entraîner vos modèles plus tard si vos besoins changent.

3. Phase 2 : ETL & Feature Engineering – La Raffinerie de l'IA

L'ETL (Extract, Transform, Load) est l'étape où vous créez de la valeur. C'est ici que l'IA intervient pour "faire parler" les données muettes.

Extract : Lecture efficace des sources multimodales depuis le Data Lake.

Transform (Le "Génie" de l'Ingénieur) :

Nettoyage & Normalisation : Gérer les valeurs manquantes ($NaN$), convertir les devises, harmoniser les formats de date.

Enrichissement (IA) : C'est le cœur du projet. Vous allez créer des Features (variables) qui n'existent pas :

Vision : Transformer une photo en un score de "Standing" (Luxe vs Standard).

NLP : Transformer un texte brut en un score de "Sentiment" ou de "Nuisance".

Load : Injection des données unifiées dans le Data Warehouse (PostgreSQL). Contrairement au Lake, le Warehouse est structuré selon un schéma (en étoile ou flocon) optimisé pour répondre aux questions métier.

4. Phase 3 : EDA & Rapport d'Impact – Le Storytelling Décisionnel

L'EDA (Exploratory Data Analysis) n'est pas juste faire des graphiques ; c'est valider ou infirmer les hypothèses du Chapitre 1.

Analyse Critique : Vous cherchez les corrélations (ex: Prix vs Sentiment). Vous détectez les Outliers (anomalies) qui pourraient révéler des fraudes massives.

Synthèse : Vous produisez un rapport de 2 à 5 pages. Le Maire ne lira pas votre code Python ; il lira vos conclusions. Chaque graphique doit appuyer une recommandation d'action.

5. L'Impératif de l'Automatisation : L'Ingénieur vs le Technicien

La grande différence entre un bidouilleur et un Data Scientist professionnel, c'est l'Automatisation.

"Si vous devez faire une tâche manuellement plus de deux fois, automatisez-la."

Votre travail consiste à créer des Pipelines reproductibles. Pourquoi ?

Fiabilité : Si les données de juillet arrivent, votre pipeline doit générer le nouveau rapport en un clic.

Scalabilité : Votre système doit pouvoir passer du quartier du Marais à la ville entière sans réécrire le code.

Auditabilité : La Mairie doit pouvoir vérifier comment vous avez calculé vos scores. Votre code est votre preuve légale.

6. La Diversité des Architectures : Choisir sa Chaîne de Valeur

Il est crucial de comprendre que la pipeline "Data Lake $\rightarrow$ ETL $\rightarrow$ Data Warehouse" est le standard industriel classique, mais ce n'est pas la seule option. Selon le projet, l'architecture peut muter :

L'approche "Full Data Lake" (Modern Data Stack) :

On stocke tout dans le Lake et on utilise des outils de calcul (comme Spark ou Presto) pour interroger directement les fichiers bruts.

Avantage : Très flexible et rapide à mettre en place.

Inconvénient : Peut devenir un "Data Swamp" (marécage de données) si personne ne documente ce qu'il y a dedans.

L'approche "Data Lakehouse" (Le meilleur des deux mondes) :

C'est la tendance actuelle. On garde la souplesse du Data Lake (stockage d'images, fichiers texte) mais on y ajoute une couche logicielle qui permet d'avoir la rigueur et la performance du Data Warehouse.

L'approche "Direct Ingestion" (Small Data) :

Pour des petits volumes, on peut parfois sauter l'étape du Data Lake et charger directement les données nettoyées dans une base SQL.

Risque : Si la source change ou si vous faites une erreur de calcul, la donnée d'origine est perdue.
