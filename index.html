Introduction & Problématique

Contexte mission

Imagine que tu es Lead Data Engineer chez UrbaTech. La ville de Casablanca te missionne pour auditer le marché immobilier. Le problème: les chiffres seuls ne disent pas tout. Il faut croiser les prix, le ressenti des clients (texte) et l'état visuel des bâtiments (images) pour identifier les zones de gentrification.

Cette mission simule une situation professionnelle complète: interpréter un phénomène urbain complexe avec des données de nature différente, puis produire une recommandation intelligible pour un décideur public.

  • Question centrale: où les signaux de gentrification sont-ils les plus visibles ?
  • Objectif pédagogique: relier ingénierie data, IA appliquée et aide à la décision.
  • Livrable final: un rapport de synthèse orienté action publique.

Sources de Données — Le Data Lake

Collecte

Vous allez constituer un Data Lake à partir de sources réelles. L'idée est de conserver les données brutes avant transformation afin de garantir traçabilité et reproductibilité.

Sources recommandées:
  • Inside Airbnb — Portail des jeux de données
  • `listings.csv` (annonces et prix)
  • `reviews.csv` (avis clients textuels)
  • Dossier d'images d'appartements (collecte locale/scraping encadré ou dataset pédagogique fourni)
Détail technique — Structure initiale du Data Lake

Exemple de dossier: /data_lake/raw/listings/, /data_lake/raw/reviews/, /data_lake/raw/images/. Conserver les fichiers horodatés pour versionner les extractions.

Phase 1 (3h) — Le Chaos Organisé

Architecture

Vous posez le socle d'architecture du projet: environnement conteneurisé, séparation des couches de stockage, et stratégie d'ingestion robuste.

Consigne 1 — Environnement Docker

Créer des services Docker pour PostgreSQL, notebook ETL, et éventuel service d'inférence.

# Exemple minimal
docker compose up -d
docker ps
Consigne 2 — Data Lake vs Data Warehouse

Data Lake: stockage brut, flexible, orienté fichiers.
Data Warehouse (PostgreSQL): stockage structuré, requêtable, orienté décision.

Consigne 3 — Stratégie d'ingestion

Définir un pipeline d'ingestion incrémental: extraction, validation de schéma, contrôle qualité, journalisation des erreurs.

Phase 2 (3h) — La Transformation Intelligente

ETL multimodal

Vous enrichissez les données pour rendre visibles des signaux faibles: sentiment des avis textuels et standing visuel des logements.

Consigne 1 — Texte vers score de sentiment

Utiliser un modèle pré-entraîné (Transformers, TextBlob, VADER selon langue/disponibilité), puis normaliser un score dans l'intervalle [-1, 1].

Consigne 2 — Image vers classe de standing

Appliquer un classifieur pré-entraîné ou une heuristique visuelle guidée pour catégoriser Luxe vs Standard.

Consigne 3 — Anticipation du schéma SQL final

Concevoir le schéma cible avant implémentation complète ETL afin de stabiliser les contrats de données. 

CREATE TABLE mart_property_signals (
  property_id            TEXT PRIMARY KEY,
  neighborhood           TEXT NOT NULL,
  price_mad              NUMERIC(12,2),
  review_sentiment_score NUMERIC(5,3),
  visual_standing_class  TEXT CHECK (visual_standing_class IN ('Luxe','Standard')),
  gentrification_index   NUMERIC(6,3),
  loaded_at              TIMESTAMP DEFAULT NOW()
);

Phase 3 (3h) — Le Verdict de l'Ingénieur

Exploration & Impact

Place à l'analyse exploratoire (EDA): comparer quartiers, détecter corrélations, interpréter des zones de tension immobilière et produire des recommandations concrètes.

Consigne 1 — EDA orientée décision

Produire des visualisations: distribution des prix, sentiment moyen par quartier, standing visuel dominant, et indice de gentrification composite.

Consigne 2 — Rapport de 2 pages pour le Maire

Structurer le rapport: contexte, méthodologie, 3 résultats majeurs, limites, recommandations actionnables (régulation, urbanisme, logement social, suivi temporel).

Consigne 3 — Argumentaire d'ingénierie

Justifier les choix techniques (modèles, métriques, architecture) et leurs impacts sur la fiabilité des conclusions.
